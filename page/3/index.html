网络编程中的读写函数

发表于 2016-06-02   |   分类于 网络编程   |  

网络编程中当连接建立好之后就是数据的传输了，系统提供了很多种读写的方式，本文对这些函数做一个大概的总结。

普通读写的流程

网络应用发送数据到对端接受到数据一般会经历下面的步骤：

  1. 系统调用read()函数产生一个上下文切换，从user mode切换到kernel mode，然后DMA执行拷培，把文件从硬盘拷培到kernel buffer里面；
  2. 数据从kernel buffer拷培到user buffer，然后系统调用read()返回，这时产生从kernal mode到user mode的切换；
  3. 然后要发送数据了，通过调用write系统调用，从user mode切换到kernel mode，然后把步骤2中user mode的数据读到kernel buffer中，这个kernel buffer是和socket相关联的；
  4. 系统调用write返回，产生一个切换，从kernel mode切换到user mode，然后DMA从kernel buffer拷培数据到协议栈。
    所以从上面可以看到，从硬盘读取数据到最终发送到网络上面总共需要四次上下文切换，然后有4次拷培操作。

TCP带外数据

发表于 2016-06-01   |   分类于 网络编程   |  

带外数据，也成为经加速的数据，是一个连接的某一端发生了重要的事情，而且希望迅速通知对端，迅速是指虽然传过去很多数据，但是需要先处理这个紧急数据，可以看做它的优先级别比较高。ftp中的ABOR命令使用带外数据来中断文件传输，telnet和rlogin会将中止字符作为紧急数据传到对端，允许对端丢弃未处理的输入，并且丢弃未发送的输出。

TCP带外数据

TCP协议并没有真正的带外数据，不过提供了紧急模式，TCP首部的URG标志位以及紧急指针偏移量两个变量。但是OOB字节的发送与否取决于发送缓冲区中先于它的字节数，window窗口大小以及TCP准备发送给对端的分节大小。下面举一个例子：
1、 发送方：发送进程已经在一个TCP套接字写入N字节数据，这些数据在套接字的发送缓冲区中，等待发送到对端，这时候，以MSG_OOB标志写入一个字节的带外数据，则该数据放在N字节的后面，并把TCP紧急指针设置为再下一个可用为值，并且将带外字节标记为OOB，下一次发送时候TCP首部会设置URG和偏移量，但是OOB呢个字节不一定发送出去（阻塞等情况，所以即使流量控制使数据无法发送，但是紧急通知照样可以发送到对端）。
2、 接收方：有新的紧急指针到达时候，接受进程的内核给套接字的属主进程发送SIGURG信号（所以接受进程要通过fcntl函数设置属主关系，而且设置信号处理函数），然后如果接受进程阻塞在select调用中，等待这个套接字的异常条件，select调用返回。一旦紧急通知到达，不论实际数据是否到达，都要通知，如果没到达，recv函数会返回EWOULDBLOCK，如果有多个OOB字节同时到达，则舍弃前面的，只保存最后一个，因为OOB字节不是放在套接字的缓冲区，而是放在该连接的一个独立的单字节的带外缓冲区，而且接收字节从这个缓冲区读取该字节的唯一方法是通过指定MSG_OOB标志调用recv,recvfrom或者recvmsg函数,当然了如果开启了SO_OOBINLINE套接字选项，这个实际数据就会在普通缓冲区中，需要检查带外标记以获得合适处理这个数据字节。

miniftp项目实战之父子进程通信-Unix域套接字

发表于 2016-05-31   |   分类于 mififtp项目实战   |  

UNIX域套接字可以用在同一台计算机上运行的进程之间的通信，虽然普通的socket也可以做这一工作，但是使用UNIX域套接字，仅仅只是复制数据，并不执行协议处理，不需要添加或者删除网络报头，无需计算校验和，序列号等封包解包的过程。域套接字存在至少有三个理由：
1、Unix域套接字由于不需要封包解包的繁杂过程，所以比普通套接字快出一倍；
2、Unix域套接字可以用来传递描述符；
3、Unix域套接字可以用于将客户的凭证（用户ID和组ID)提供给服务器，提供额外的安全检查措施。

UNIX域套接字地址结构

无名域套接字

可以使用面向网络的域套接字接口或者socketpair建立一堆无命名的，相互连接的域套接字。

这样一对域套接字相当于一个全双工的管道，两端对读写都开放.socketpair创建的这两个描述符，不属于任何的实际文件系统，而是网络文件系统，虚拟的．同时内核会将这两个描述符彼此设为自己的peer即对端（这里即解决了如何标识读写端，可以想象，两个描述符互为读写缓冲区，即解决了这个问题）．然后应用相应socket家族里的read/write函数执行读写操作．

传递文件描述符

我们经常会有一个进程向另一个进程传递文件描述符的需求，可能因为一个进程有文件系统权限，而另外一个没有(ftp服务器nobody进程)；也可能因为服务器讲一个连接的文件描述符传递给另外一个已经启动的协助进程等情况，一个进程向另一个进程传递文件描述符使得使用文件描述符的进程对打开文件或者设备的所有细节都是透明的！
从一个进程向另一个进程传递文件描述符可以有两种方法，一种是通过fork来进行父子进程的文件描述符传递，另外就是靠UNIX域套接字。
